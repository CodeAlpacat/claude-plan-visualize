📋 PLAN

이메일 + 비밀번호 로그인

현재 소셜 전용인 로그인에 이메일 옵션을 추가해, 소셜 화면에서 이탈하는 15% 유저를 회복.

복잡도: ●○○○○ Simple 파일: 6개 (NEW 5 / EDIT 1) 새 테이블: 1개 API: 2개

🎯 Why

내부 분석상 가입 시도의 15% 가 소셜 제공자 화면에서 이탈 — 3rd-party 권한 부여 원치 않거나 해당 계정 없는 유저. 이메일/비밀번호 추가로 이탈 대부분 회복 가능.

⚠️ 리스크

🚨 P0 — 로그인 지연 spike

bcrypt cost 너무 높음 → 유저가 앱 느리게 인지 → 이탈.

대응: cost=10, staging 에서 p95 지연 측정 후 출시.

⚠️ P1 — 비밀번호 재설정 흐름 없음

비밀번호 까먹은 유저 막힘.

대응: "비밀번호 찾기? 고객센터 문의" 임시 링크. 재설정 흐름은 다음 PR.

⛔ 이번 PR 안 함

  • 2FA / TOTP — 다음 PR
  • 매직 링크 — backlog
  • 비밀번호 재설정 — 다음 PR (임시: 고객센터 이메일)
  • Rate limiting — 별도 보안 PR

🔄 사용자 흐름

sequenceDiagram
  actor U as 유저
  participant UI as 로그인 화면
  participant API as /api/auth/login
  participant DB as auth_passwords
  participant Sess as 세션

  U->>UI: 이메일 + 비밀번호 입력
  UI->>UI: zod 검증 (이메일 형식, 비밀번호 8자+)
  UI->>API: POST { email, password }
  API->>DB: SELECT by user_id
  DB-->>API: hash 행
  API->>API: bcrypt.compare(password, hash)
  alt 일치
    API->>Sess: 세션 쿠키 set
    Sess-->>API: ok
    API-->>UI: 200 { user, session }
    UI-->>U: /dashboard 리다이렉트
  else 불일치
    API-->>UI: 401 { error }
    UI-->>U: 인라인 "이메일 또는 비밀번호가 일치하지 않습니다"
  end

🗃️ 데이터 모델

erDiagram
  USERS ||--o| AUTH_PASSWORDS : "1:1 optional"
  AUTH_PASSWORDS {
    uuid user_id PK_FK
    text hash
    timestamptz created_at
    timestamptz updated_at
  }

user_id 는 PK + FK 동시. ON DELETE CASCADE 로 유저 삭제 시 password 행 자동 정리. Optional 1:1 — 소셜 전용 유저는 행 없음.

🔌 API

POST/api/auth/signup

입력: { email, password }

출력: 201 { user, session } · 409 — 이메일 중복

POST/api/auth/login

입력: { email, password }

출력: 200 { user, session } · 401 — 불일치
